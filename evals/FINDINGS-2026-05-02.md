## Issues surfaced by running the pipeline

### Issue 1 — Extract is effectively unusable without `GITHUB_TOKEN`
- 9 seed skills × 3 fetches (SKILL.md + claude.yaml + openai.yaml) = ~27 requests, dispatched at concurrency 8.
- Anonymous limit is 60/hour; the budget is exhausted within seconds.
- `GitHubClient._maybe_sleep_for_quota` then sleeps each in-flight task for ~50 minutes (`remaining<50` and the next reset is far in the future).
- The sleep happens **before** the 4xx is converted to `GitHubClientError`, so tasks that are about to hard-fail still wait the full ~50 minutes first.
- Two consecutive extract runs both hung; only 3 of 9 skills fetched a `SKILL.md` (the rest got 403/404), and the run never completed.

### Issue 2 — No fail-fast or pre-flight for missing token
- README and `evals/cli.py` quickstart say `GITHUB_TOKEN` is required, but the CLI emits a single warning and proceeds.
- Surface area of the silent-warning bug is large because the next 50-minute wait is invisible to the operator (only visible at `--log-level DEBUG`/INFO).
- Suggest: error out unless `--allow-anonymous` is passed, and skip seeds whose budget projection would exceed the remaining quota.

### Issue 3 — `report` does not (re)build projections
- After a successful `eval` step, `evals_current`/`skills_current` are empty until `project --rebuild` is invoked manually.
- The README quickstart goes `discover → extract → eval → report` and silently produces an empty table:
  `[OK] report rendered: skills=0 S=0 A=0 B=0 C=0 D=0`.
- The reporter should fold projections in, or report a clear "projections stale, run `evals project`" diagnostic.

### Issue 4 — Phase-1 only reliably caps every skill at tier D
- Composite score weights all four buckets (`health`, `provenance`, `compliance`, `security`).
- With only phase-1 evaluators wired up (`compliance-check`, `security-scan`), the `health` and `provenance` buckets are always 0.0.
- A perfectly compliant local skill therefore tops out at ~45/100 and falls in tier D — same tier as a skill we never extracted.
- Composite scoring should either normalize over present buckets, or the tier mapping should account for "missing-phase" vs "failed-phase".

### Issue 5 — False positive: `allowed-tools` "wildcard" fires on scoped Bash patterns
- `evals/evaluators/compliance_check.py:33` flags any `*` in `allowed-tools` as a wildcard.
- `plugins/md-export/skills/export-pdf/SKILL.md` uses `allowed-tools: Bash(node:*) Read Glob` — the **recommended** scoped form per CLAUDE.md ("must explicitly enumerate every tool (no wildcards)").
- The compliance check conflates `Bash(node:*)` (an argument-scoped pattern that *narrows* permissions) with bare `*` or `Bash(*)` (true wildcards that *widen* them).
- Suggest: detect specifically `\*` outside of `(...)`, or recognized true-wildcard tokens (`*`, `Bash(*)`, `Bash:*`, `WebFetch(*)`).

### Issue 6 — Duplicate / inconsistent finding strings across evaluators
- `compliance-check` emits `"allowed-tools missing"`; `security-scan` emits `"allowed-tools missing for scope check"`. Same root cause counted twice.
- Wildcards likewise counted twice: `"WARN: allowed-tools contains wildcard; should enumerate explicit tools"` (frontmatter parser) **and** `"allowed-tools contains wildcard"` (compliance check) **and** `"allowed-tools wildcard widens scope"` (security scan) — three findings for one input.
- Reduces signal/noise of the report; suggest a shared finding code (`AT001-missing`, `AT002-wildcard`) so dedup is trivial.
